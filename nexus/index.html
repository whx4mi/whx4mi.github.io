NEXUS v2.0
OSINT Intelligence Platform · Recon Passivo · BR Intel
CONCLUÍDO LABORATÓRIO
Plataforma OSINT single-file que centraliza reconhecimento passivo de alvos em uma interface unificada. Em vez de executar dorks manualmente no Google e consultar APIs separadas, o NEXUS gera automaticamente dorks contextuais por tipo de entidade (domínio, empresa, pessoa, IP) e agrega fontes de inteligência — incluindo fontes brasileiras específicas.

Usado em consultorias freelance para fase de reconhecimento passivo antes de qualquer interação com o alvo, com exportação de relatório HTML para entrega ao cliente.
linguagem Python 3 arquitetura single-file uso real consultorias freelance
// Módulos principais
Gerador de Dorks (145+)
Geração contextual de Google Dorks baseada no tipo de entidade alvo. Cada categoria de dork tem lógica própria — não é lista estática, é template engine contextual.
Entity Risk Scoring
Cálculo automático de score de risco do alvo baseado em presença de dados expostos, breaches encontrados e footprint digital. Output quantitativo para priorização.
HIBP Integration
Consulta à API Have I Been Pwned para verificação de emails e domínios em breaches conhecidos. Retorna lista de vazamentos com data e dados comprometidos.
BR-INTEL Tab
Módulo específico para fontes de inteligência brasileiras — consulta de CPF/CNPJ em fontes abertas, registros de domínio .br e bases de dados públicas brasileiras.
HTML Report Export
Geração de relatório HTML formatado com todos os findings, score de risco e dorks organizados por categoria — pronto para entrega em consultorias.
Multi-entity Support
Suporte a diferentes tipos de alvo: domínio, empresa, pessoa física, endereço IP. Cada tipo gera dorks e consultas específicas ao contexto.

// O que aprendi
  • OSINT como metodologia — recon passivo tem etapas definidas: footprint → exposure → breach intel → scoring. Colocar isso em código força clareza sobre o processo.
  • APIs de inteligência — integração com HIBP exigiu entender rate limiting, autenticação e parsing de respostas estruturadas. Base para integrar outras APIs de threat intel.
  • Contexto brasileiro específico — Brasil tem fontes de dados abertas únicas (Receita Federal, Registro.br, DOU) que ferramentas internacionais ignoram. Explorar isso é diferencial em consultorias locais.
  • Risk scoring — traduzir dados qualitativos (tem breach? tem admin exposto? tem credenciais no GitHub?) em score quantitativo exigiu pensar em pesos e normalização.
  • Relatório como entregável — gerar HTML bem formatado para cliente não técnico é diferente de output de terminal. Aprendi a separar coleta de dados de apresentação.
